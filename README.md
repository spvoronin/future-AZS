# ⛽ Цифровая экосистема АЗС (Gas Station ERP & Core API)

## 📝 Описание концепта проекта
Этот проект представляет собой **высокофункциональную бэкенд-платформу (Core API)** для автоматизации и управления сетью автозаправочных станций. В реальном секторе (Enterprise) подобные системы являются цифровым сердцем топливных гигантов. Они связывают воедино мобильные приложения клиентов, CRM-системы управления бизнесом и физическое промышленное оборудование (контроллеры ТРК, датчики резервуаров) на самих станциях.

Вместо использования абстрактных ORM (вроде SQLAlchemy), проект реализован на **асинхронном FastAPI** и **чистом SQL (через psycopg2)**. Это обеспечивает максимальный контроль над транзакциями, оптимизацию запросов к базе данных и минимальный оверхед (low overhead) при высоких нагрузках.

## 📐 Архитектура сущностей и связей (Текущее состояние)

На данном этапе архитектура приложения разделена на три логических слоя, которые тесно связаны на уровне базы данных:

```text
[Пользователи / Водители] (users)
│
▼
[Автозаправочные станции] (stations) ───◄ [Связаны один-ко-многим]
│
▼
[Топливные колонки / ТРК] (pumps)
```

### 1. 👤 Модуль пользователей (`router_users`)
Отвечает за управление цифровыми профилями клиентов АЗС.
* **Бизнес-логика:** Помимо стандартной аутентификации (email/пароль), в профиль зашит `number_of_car` (госномер автомобиля). В перспективе это позволяет интегрировать систему с умными камерами на въезде АЗС для автоматического распознавания клиента и предложения его «любимого» топлива.
* **Безопасность запросов:** Процесс регистрации (`/register`) оптимизирован с помощью конструкции `RETURNING id`, что гарантирует получение корректного ID пользователя при высоких параллельных нагрузках.

### 2. 🏢 Модуль станций (`router_stations`)
Физический каркас распределенной сети заправок.
* **Бизнес-логика:** Каждая станция имеет географическую привязку (`region`, `adress`) и коммерческий маркер (`rating`).
* **Агрегация данных:** Эндпоинт `GET /stations/{id}/pumps` выступает связующим звеном, позволяя динамически выгружать всё оборудование (колонки), закрепленное за конкретной АЗС.

### 3. 🔌 Модуль топливных колонок ТРК (`router_pumps`)
Слой управления исполнительным оборудованием на станциях.
* **Бизнес-логика:** Каждая ТРК имеет статус работы (`status`) и флаг доступности (`is_active`). Этот модуль — самый динамичный, так как состояние ТРК меняется в реальном времени при каждом цикле заправки.
* **Отказоустойчивость:** Обновление параметров (`PUT`) защищено явными проверками на `None` (для булевых типов), что исключает логические баги при переключении активности колонки фронтендом.

## 🔮 Вектор развития: Что превратит этот проект в мощный бизнес-кейс?

Поскольку проект активно развивается, заложенная архитектура FastAPI + PostgreSQL позволяет масштабировать его в полноценную экосистему через следующие шаги:

### Шаг 1: Модуль нефтепродуктов (`fuels` & `reservoirs`)
* **Сущность Топлива:** Добавление таблицы типов топлива (АИ-95, АИ-92, Дизель, Пропан, Электрозарядка) со стоимостью за 1 литр/кВт.
* **Резервуары (Складской учет):** Создание логики мониторинга объемов топлива в подземных цистернах станций. Бэкенд должен отслеживать остатки и при достижении критического минимума (например, < 1000 литров) автоматически генерировать заявку для бензовозов.

### Шаг 2: Модуль коммерции и транзакций (`orders`)
Сердце бизнес-логики, связывающее `user_id`, `station_id` и `pump_id`:
1. **Проверка состояния:** Бэкенд проверяет, активна и свободна ли колонка.
2. **Блокировка ресурса:** Изменение статуса ТРК на "Занята/Налив", чтобы предотвратить одновременную оплату одной колонки двумя клиентами.
3. **Финансовый расчет:** Калькуляция стоимости заказа по формуле: $\text{Объем (л)} \times \text{Цена за литр}$.
4. **Жизненный цикл заказа:** Проведение транзакции через статусы: `Создан` ➔ `Оплачен` ➔ `Налив` ➔ `Завершен`.

### Шаг 3: Асинхронные фоновые процессы (`Background Tasks`)
Процесс физического налива топлива занимает время (например, 1-2 минуты). Бэкенд не должен блокировать основной поток выполнения (event loop) в ожидании окончания заправки.
* Реализация фоновых задач (`BackgroundTasks` в FastAPI или брокер `Celery`), которые будут имитировать процесс заправки, изменять статус ТРК в реальном времени и автоматически закрывать ордер по завершении налива.

## 🎯 Заключение
Выбранная предметная область (АЗС) является отличным примером сложной распределенной системы. Проект не ограничивается простым выводом текста на экран, а решает реальные индустриальные задачи: конкурентный доступ к ресурсам, консистентность данных при транзакциях и управление состояниями оборудования.
